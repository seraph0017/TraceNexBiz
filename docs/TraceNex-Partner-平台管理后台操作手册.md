# TraceNex Partner 平台管理后台操作手册

> 适用对象：平台运营、财务、KYC 审核、客服、风控/合规人员。
> 测试后台地址：`https://admin-test.tracenex.cn`
> 说明：本文按当前 TraceNex Partner 管理后台菜单编写。测试环境里部分模块可能还处于联调或空数据状态，最终以页面按钮和接口返回为准。

## 1. 这个后台是干什么的

TraceNex Partner 是分销/渠道商平台。它不负责 AI 模型调用本身，AI 网关能力在 Fy-api 里；Partner 平台主要负责：

- 管渠道商：创建、查看、终止合作。
- 管实名/KYC：审核渠道商或客户提交的身份资料。
- 管钱包和调账：查看渠道商钱包余额，做人工调账。
- 管结算：按账期生成渠道商应付金额，锁定、下发、对账。
- 管退款和发票红冲：处理退款申请、处理错误发票的冲销。
- 管合规：审计日志、内容安全上报、PIA 报告、PIPL 投诉。
- 管系统参数和员工账号。

## 2. 登录与退出

1. 打开 `https://admin-test.tracenex.cn`。
2. 输入平台员工账号、密码。如页面要求动态口令，输入 MFA/OTP。
3. 登录后默认进入“渠道商”页面。
4. 右上角用户头像菜单可退出当前会话，或全设备登出。

注意：

- 后台是员工系统，不要把账号借给渠道商或客户。
- 财务、员工管理、强制处理 Saga 等敏感操作应由对应角色执行。
- 如果页面提示 IP 不在白名单，需要联系管理员调整安全参数。

## 3. 渠道商

入口：`渠道商`

这个页面用于管理分销渠道商。

### 新建渠道商

1. 点击“新建渠道商”。
2. 填写：
   - Name：渠道商联系人或公司显示名。
   - Email：联系邮箱。
   - Phone：手机号。
3. 点击确定。
4. 新建成功后，列表会出现该渠道商。

说明：

- 当前后台新建会先生成一个渠道商记录，用于测试和后续 KYC/结算流程。
- 正式生产里，渠道商一般还需要完成 KYC、签约、收款账户配置，才适合放量。

### 查看渠道商详情

点击列表里的 `#ID` 进入详情页，可查看：

- 状态
- 联系邮箱、手机号
- 客户数量
- 月度毛收入、净收入
- 终止时间、宽限期截止时间

### 终止合作

在详情页点击“终止合作”，填写终止原因后提交。

终止后系统会进入 30 天客户宽限期：该渠道商名下客户需要在宽限期内迁移或重新选择渠道商，避免服务突然中断。

## 4. KYC 审核是什么

入口：`KYC 审核`

KYC 是 “Know Your Customer / Know Your Client”，中文可以理解为“实名审核、主体审核”。目的不是走形式，而是确认平台服务对象是谁，避免给冒名、虚假主体、违规主体开通资金和分销能力。

在本平台里，KYC 可能针对：

- 渠道商：公司/个人代理身份、营业执照、法人信息、收款账户等。
- 客户：需要实名的终端客户场景。

### KYC 审核看什么

列表页展示：

- 对象类型：partner 或 customer。
- 名称。
- 状态。
- 创建时间。

详情页展示：

- 实名姓名或主体名称。
- 证件号脱敏信息。
- 上传材料链接。
- 三方核验结果。

### 三方核验是干嘛的

“调用三方核验”一般是调用外部 OCR、工商、身份证、银行卡或实名服务，检查用户提交的材料是否真实、匹配。

例子：

- 公司名和营业执照是否匹配。
- 法人姓名和证件号是否匹配。
- 银行账户是否像一个合法收款账户。

### 怎么审核

1. 进入 `KYC 审核`。
2. 点击某条记录的 `#ID`。
3. 先看资料和三方核验结果。
4. 如果资料真实且符合准入标准，点击“通过”。
5. 如果资料不完整、明显不匹配或有风险，点击“驳回”，填写：
   - 驳回代码：方便后续统计，例如 `MISSING_LICENSE`、`ID_MISMATCH`。
   - 驳回详细原因：给运营/用户看的说明。

建议：

- 不要只因为“资料看起来像真的”就通过，高风险主体必须结合三方核验。
- 驳回原因要写清楚，方便对方补材料。
- 涉及身份证、营业执照、银行卡等材料时，不要下载后随意转发。

## 5. 钱包与调账

入口：`钱包`

钱包用于记录渠道商相关的资金余额或应付余额。列表页展示渠道商、余额、更新时间。

### 调账

入口通常是钱包调账页面。

填写：

- Partner ID：渠道商 ID。
- 金额：单位是元。
- 原因：必须写清楚，会进入审计日志。

调账适合处理：

- 测试环境补余额。
- 人工修正异常余额。
- 财务确认后的补偿或扣减。

注意：

- 调账是高风险操作，必须写原因。
- 不要用调账替代正常充值、结算、退款流程。

## 6. 结算

入口：`结算`

结算是按账期统计渠道商应得金额，并推动分账/付款。

核心字段：

- 账期：例如 2026-05。
- 渠道商数：本批次包含多少渠道商。
- 总额：本批次合计金额。
- 状态：draft、locked、dispatched、settled。

### 结算流程

1. draft：草稿态，金额还可以重算。
2. locked：锁定批次，金额进入确认态，不应随意变动。
3. dispatched：分账下发，提交给持牌分账方或支付/结算通道。
4. settled：回执对账完成。

### 锁定批次

在结算详情页点击“锁定批次”。锁定后表示财务认可该批次数据，可以准备下发。

### 分账下发

在 locked 状态点击“分账下发”。这一步会进入真实或模拟的付款/分账链路。

### 回执对账

下发后拿到支付通道、持牌方或银行的回执 ID，填写“回执 ID”完成对账。

注意：

- 结算前要确认渠道商 KYC、收款账户、税务信息是否完整。
- 页面提示“代扣个税”表示系统会计算税额，实际代扣规则需以财务/税务配置为准。

## 7. 退款

入口：`退款`

退款用于把已经发生的充值、结算或其他来源金额退回或冲正。

### 发起退款

填写：

- Origin：来源类型，通常是 topup 或 settlement。
- Origin ID：原始业务记录 ID。
- 金额：单位元。
- 原因：退款原因。

### 审核退款

列表中状态为 submitted 或 reviewing 的退款，可以审核通过或驳回。

审核通过后，系统会触发退款 Saga。

### Saga 是什么

Saga 可以理解为“跨系统事务流程”。退款不是改一个数据库字段就完事，它可能要同时处理：

- Partner 平台退款记录。
- 钱包余额。
- Fy-api 额度。
- 支付/分账通道。
- 审计日志。

这些步骤任何一步失败，都需要重试、补偿或转人工。所以后台有 Saga 相关功能。

## 8. 红冲是什么

入口：`红冲`

红冲是发票领域的词，全称常叫“红字发票冲销”。白话说：之前开出去的发票有问题，不能直接删除，就要开一张负数/红字发票把它冲掉。

常见场景：

- 金额开错了。
- 抬头开错了。
- 业务发生退款，需要冲销原发票。
- 原发票需要作废但已经不能直接作废。

### 为什么不能直接删发票

发票是税务凭证，已经开具的发票要留下完整链路。正确做法是：

1. 保留原发票。
2. 发起红冲申请。
3. 审核红冲原因。
4. 开具红字发票或完成税局要求的红冲流程。
5. 审计日志记录整个过程。

### 怎么处理红冲

1. 进入 `红冲`。
2. 查看状态为 applying 的记录。
3. 点击“审核”。
4. 填写原因代码和原因说明。
5. 提交。

建议原因代码示例：

- `AMOUNT_ERROR`：金额错误。
- `TITLE_ERROR`：发票抬头错误。
- `REFUND`：退款导致冲销。
- `BUSINESS_CANCELLED`：业务取消。

## 9. 审计日志

入口：`审计日志`

审计日志记录后台关键操作，包括操作者、动作、目标、结果、IP、Trace ID。

可以按动作或操作者过滤。

### 哈希链校验是什么

审计日志使用类似链条的方式保存：后一条记录会带着前一条记录的哈希。这样如果有人中途改了日志，链条就会断。

点击“校验哈希链”可以检查日志是否被篡改。

## 10. 内容安全 12377

入口：`内容安全 (12377)`

这个模块用于处理平台发现的违法或不良内容上报队列。

12377 是中国互联网违法和不良信息举报中心。平台如果检测到严重违规内容，需要按规则留痕、处理，必要时上报。

页面能做：

- 查看待上报/已上报/失败的内容安全记录。
- 查看内容摘要和详情。
- 对失败记录重试。
- 一键派发 pending 队列。

注意：

- 详情页可能包含敏感文本，只允许授权人员查看。
- 不要把用户输入、违规内容截图随意外传。

## 11. PIA 报告是什么

入口：`PIA 报告`

PIA 是 Privacy Impact Assessment，中文一般叫“个人信息保护影响评估”。

白话说：当平台要处理个人信息，尤其是敏感个人信息、自动化决策、向第三方共享数据、跨境/委托处理等场景，需要评估“这么做会不会伤害用户隐私、有没有合规风险、风险怎么控制”。

PIA 报告通常回答：

- 我们处理什么个人信息。
- 为什么要处理。
- 处理范围是否必要、是否最小化。
- 会共享给哪些第三方。
- 保存多久，什么时候删除。
- 有没有加密、脱敏、访问控制、审计。
- 如果用户投诉或要求删除，怎么处理。

### 生成报告

1. 进入 `PIA 报告`。
2. 点击“生成新报告”。
3. 输入评估范围，例如：
   - `KYC 文件上传与审核`
   - `渠道商结算与发票`
   - `内容安全检测与 12377 上报`
4. 提交后列表出现报告。
5. 如果有下载链接，可下载归档。

注意：

- PIA 不是运营日报，而是合规评估材料。
- 正式环境里，重大功能上线前应该先有对应 PIA 或合规评估记录。

## 12. PIPL 投诉处理

入口：`PIPL 投诉`

PIPL 是《中华人民共和国个人信息保护法》。PIPL 投诉就是用户围绕个人信息权益提出的投诉或请求。

常见类型：

- export：导出个人信息。
- delete：删除个人信息。
- rectify：更正个人信息。
- portability：转移个人信息。

### 处理投诉

1. 进入 `PIPL 投诉处理`。
2. 点击投诉 ID 查看详情和时间线。
3. 根据实际处理结果选择：
   - 已处理
   - 驳回
4. 填写备注。
5. 提交。

注意：

- 不要随意驳回用户的个人信息权利请求。
- 涉及删除、导出、修改时，要确认申请人身份。
- 处理过程要留痕。

## 13. 安全参数

入口：`安全参数`

用于配置后台安全策略：

- IP 白名单：限制哪些 IP 可以访问后台。
- Step-up MFA TTL：敏感操作二次验证有效期。
- 密码策略。
- 屏幕水印开关。
- 会话最长时间。

建议：

- 生产环境必须开启 IP 白名单和 MFA。
- 水印建议开启，便于截图泄露追责。
- 不要随意放宽会话时长。

## 14. 业务参数

入口：`业务参数`

用于维护平台业务配置，例如：

- 备案号。
- 模型白名单。
- 定时任务配置。
- 合规联系电话/链接。
- 发票、支付、内容安全相关开关。

操作方式：

1. 找到要改的 key。
2. 点击编辑。
3. 修改 value。
4. 保存。

注意：

- 业务参数可能直接影响线上流程。
- 修改前应确认 key 的含义，必要时先在测试环境验证。

## 15. Saga force-resolve 是什么

入口：`Saga force-resolve`

这是异常处理工具，不是日常运营按钮。

当退款、调账、分账等跨系统流程卡在未知状态，自动重试也无法确认结果时，系统会把 Saga 标记为 escalated，需要人工介入。

force-resolve 就是人工强制把 Saga 标记为：

- resolved：确认已经成功。
- compensated：确认已补偿或回滚。

为了防误操作，这里要求 dual-control：

- 需要另一个员工生成一次性 approver token。
- 审批者不能和提交者是同一个人。
- 还有冷却时间、IP 子网等限制。
- 全程进入审计日志。

只在技术/财务共同确认结果后使用。

## 16. 员工管理

入口：`员工管理`

用于创建、查看、停用后台员工账号。

角色示例：

- super_admin：超级管理员。
- risk_admin：风控管理员。
- finance_admin：财务管理员。
- cs_admin：客服管理员。
- kyc_reviewer：KYC 审核员。

### 新建员工

填写用户名、一次性密码、角色、邮箱后提交。

建议：

- 按最小权限分配角色，不要所有人都给 super_admin。
- 离职或岗位变动后及时停用账号。
- 敏感角色应启用 MFA。

## 17. 日常运营建议流程

### 新渠道商入驻

1. 新建渠道商。
2. 要求渠道商提交 KYC 材料。
3. KYC 三方核验。
4. 审核通过。
5. 配置钱包、收款账户、结算参数。
6. 小流量测试。
7. 正式放量。

### 月度结算

1. 查看结算批次。
2. 抽查渠道商收入、成本、税额、应付。
3. 锁定批次。
4. 分账下发。
5. 拿到回执后对账。
6. 归档发票、回执和审计记录。

### 退款/红冲

1. 确认原始业务单据。
2. 发起退款。
3. 财务审核。
4. 退款 Saga 完成后检查状态。
5. 如涉及发票，发起红冲。
6. 保存原因和审计记录。

### 合规巡检

1. 定期查看 KYC 待审和驳回原因。
2. 检查内容安全失败上报。
3. 定期生成或更新 PIA 报告。
4. 处理 PIPL 投诉。
5. 校验审计日志哈希链。

## 18. 常见状态解释

### 渠道商状态

- active：正常。
- suspended：暂停。
- terminating：终止处理中。
- terminated：已终止。

### KYC 状态

- submitted：已提交，待审核。
- approved：已通过。
- rejected：已驳回。
- frozen_yearly_limit：年度驳回次数达到限制，暂时冻结。

### 退款状态

- submitted：已提交。
- reviewing：审核中。
- approved：审核通过。
- rejected：审核驳回。
- executing：执行中。
- completed：已完成。

### 结算状态

- draft：草稿。
- locked：已锁定。
- dispatched：已下发。
- settled：已对账完成。

### 内容安全状态

- pending：待派发。
- dispatching：派发中。
- submitted：已提交上报。
- failed：失败，需重试或人工处理。

## 19. 高风险操作清单

以下操作不要随便点：

- 终止渠道商。
- 钱包调账。
- 锁定和下发结算。
- 审核通过退款。
- 发票红冲。
- Saga force-resolve。
- 修改安全参数。
- 新建或停用员工。

执行这些操作前，至少确认：

- 操作对象 ID 是否正确。
- 金额是否正确。
- 原因是否写清楚。
- 是否需要第二人复核。
- 是否会影响客户服务或真实资金。
