## TL;DR

v0.2 在合规面是**实质性整改**，不是文字搬运：

- 二清问题被**结构性**解掉了（§7.6 重写 + `partner_wallet` 语义重定义 + §15.2 规则）。这是本轮整改最有质量的一项。
- 生成式 AI 提供者义务从"不做"改为"M12-01~07 + §15.3"完整内容安全模块。点名了国产合规审核服务（阿里云内容安全 / 腾讯天御），不再是空头承诺。
- 个人渠道商代扣代缴税率表**经核对正确**（劳务报酬累进与速算扣除数正确，与 41 号公告合并使用没有错位）。
- ICP / 等保 / PIA / CAC 标准合同 / 全电发票 / 单独同意 / 5 年冷归档全部进入 §15 时间表，并在 §11、§12.0、§12.5 三处交叉引用，可执行性可信。

但有 **3 项 PRE_LAUNCH** 需要在 Phase 2 hard-gate 前补齐（不阻塞 v1.0 文档），以及 **1 项 MONITOR**：

1. **ICP 经营许可证的前置条件未完整披露**——v0.2 只点了 "注册资本 100 万实缴"（Q11），没把 "1 年经营 / 30 万社保（部分省市） / 一年内审计报告" 一并写入风险条目。这会导致 Week 0 启动后才发现卡死。
2. **算法备案 + 违法内容上报通道**只写了"构建"——没说由谁备案、备案路径、上报字段、应急响应 SLA。M12-04 / M12-06 / Q14 三处都把这件事推到后续。
3. **Cookie 同意 + DPO 公示入口**——附录 A 隐私政策列了 cookie 章节、§15.1 任命了 DPO，但 §7 产品需求里没有 cookie banner / DPO 联系入口的实现条目，附录 A 不会自己变成产品。
4. **(MONITOR)** Pub/Sub & outbox 跨境数据归属未声明——v0.2 §6.3 / §5.1 没明示 Redis / outbox 实例的物理 region，SG 启用后会触发出境识别问题。

剩下的（深度合成水印、广告法极限词、CIIO 监控、PDPA）维持 Round-1 的 POST_LAUNCH / MONITOR 等级。

## Round-1 BLOCK_LAUNCH 整改核验（4/4 通过）

### BLOCK #1 二清 / 无证清算 — ✅ **真实解掉，不是文字游戏**

**核验点 1：资金流是否物理上不再过平台账户？**

§7.6 资金流图（v0.2 line 540-549）：
```
客户付款 → 持牌分账方收单 → 持牌分账方按指令分账
```
明确写"客户钱**不进** TraceNex 的台账"（line 551）。这是结构性重设，不是粉刷。

**核验点 2：`partner_wallet` 语义是否重定义？**

§8.3（line 689-702）首行注释："这不是'为渠道商保管的客户付款'，而是'平台应付给渠道商的累积分润负债'"。

字段 `Balance` 注释："累积应付分润（quota 单位）" — 即**应付台账负债**。

§15.2 进一步明确："`partner_wallet.balance` = '持牌方账面上、平台应付而未支付给渠道商的累积金额台账'，**不是**沉淀的客户付款"。

→ 语义切换到位。这正是 Round-1 Required: 从"客户付款沉淀"→"应付台账负债"。

**核验点 3：客户钱包概念是否消失？**

§7.6 line 553："客户的可用额度还是 `User.quota`（Fy-api 侧），**没有** TraceNexBiz 的客户钱包概念"。

### BLOCK #4 个人渠道商代扣代缴 — ✅ **税率表正确**

**核验点 2：税率表是否正确？**

§15.4 line 1633-1638，对照《个人所得税法》及实施条例（劳务报酬，居民个人综合所得不预扣除外条款）：

| 收入区间（元） | v0.2 公式 | 标准公式 | 一致性 |
|---|---|---|---|
| ≤ 800 | 不扣 | 减除费用 800，不扣 | ✅ |
| 800-4000 | (收入-800) × 20% | (收入-800) × 20% | ✅ |
| 4000-25000 | 收入 × 80% × 20% | (收入×(1-20%)) × 20%，20% 档（≤2万应纳税所得额） | ✅ |
| 25000-62500 | 收入 × 80% × 30% - 2000 | 30% 档（2万-5万），速算扣除数 2000 | ✅ |
| > 62500 | 收入 × 80% × 40% - 7000 | 40% 档（>5万），速算扣除数 7000 | ✅ |

边界点 25000 → 应纳税所得额 = 25000×0.8 = 20000（即 20% 与 30% 档分界）✅
边界点 62500 → 应纳税所得额 = 62500×0.8 = 50000（即 30% 与 40% 档分界）✅

**速算扣除数 2000 / 7000 与劳务报酬预扣预缴一致。**

### NEW-4 (MONITOR) Pub/Sub & outbox 数据归属未声明

- §5.1 / §6.3 / §C-6 提到 Redis Pub/Sub + outbox，但**没明示 Redis / outbox 实例物理 region**。
- 一旦 SG 站点启用，若 SG / CN 共享 Redis 或跨区订阅 Pub/Sub，**事实上构成出境**。
- consume_log_outbox 含 `fy_user_id` + `quota` + `model`——`fy_user_id` 在 PIPL 下属于一般 PI（不是敏感 PI），但跨境复制仍触发 CAC 标准合同 / 安全评估义务。

**Required**：在 §15.7 "跨境数据"部分追加：
- "Redis / outbox 实例必须 region-isolated：CN 实例的 outbox / Redis 不允许 SG 实例订阅，反之亦然。"
- "跨实例数据复制（仅在数据迁移/灾备场景）须经 DPO 审批 + 走 CAC 备案。"

进入 MONITOR 等级，因为 v1.0 期 SG 还没启用，但属于"启用前需要解锁"的硬条件。

### NEW-5 (POST_LAUNCH) 反洗钱可疑交易报告主体认定

- §15.1 列了 "反洗钱客户身份识别制度"（Week 4-8），但未列 "可疑交易报告"。
- 2025 修订版《反洗钱法》将"特定非金融机构"的覆盖范围扩展到部分支付场景。TraceNex 名义上把支付义务全外推给持牌方，但作为生成式 AI + 含大量 PII + 月流水可观的平台，仍可能被认定为"协助义务主体"。
- 这是**监管态度问题**，目前不阻塞。

**Required**（POST_LAUNCH）：上线 6 个月后 review 是否需要建立内部 STR（Suspicious Transaction Report）流程；若被监管约谈则升级。
